# スライド構成

このファイルの位置づけ:

- **`index.md`**: 本番スライド本体
- **スピーカーノート（`<!-- -->`）**: 当日読み上げる/参照する補助テキスト
- **`_notes.md`**: 外部 AI で起こした初版の素案（叩き台）
- **本ファイル `_slide_structure.md`**: 「なぜこの構成か」を考えるための作業場
- **汎用フレームワーク**: スライド構成の一般的な考え方は [.claude/storytelling/20-slides.md](../../../.claude/storytelling/20-slides.md) に抽出。本ファイルはその個別デッキへの適用例

スピーカーノートに長文を入れると本番中に読み始めて脳のリソースを持っていかれるので、設計意図・検討中の選択肢・削った理由はここに置く。スライド完成後も「なぜこうしたか」の記録として残す。

## 0. 登壇形式の制約（スピーカーガイドより）

- **アスペクト比**: 16:9（不可なら 4:3 でも可）
- **フォントサイズ**: 18pt 以上が望ましい
- **文字量**: なるべく少なく

設計への影響:

- 18pt 縛り + 文字量最小化 → 1 枚で説明しきろうとせず、概念単位で分割する方針
- コードスニペット（セクション 3 の Locust `@task`）はフォントが小さくなりがち。**3〜5 行に絞る**前提で設計
- 各スライドに長い解説を載せず、補助テキストはスピーカーノートに退避（ただし読み上げない量に抑える）

## 1. 物語の幹（これが抜けたら話が崩れる）

> **「わたしがやっていることは、紛れもなく SRE だった」**

辿り着くまでの最小構成:

1. **きっかけ** — Cloud SQL の突発高負荷。**スケールアップは選びにくく**（ダウンタイムを伴う + コスト圧 + その他諸々）、限界性能を自分で測る必要があった
2. **行動** — Locust で負荷試験を始める。自前テレメトリでメトリクスを積み上げる
3. **発見** — アプリ側も `FIXME` で問題を認識していた。負荷試験は「暴く」道具ではなく**改善効果を検証する**道具だった
4. **転換** — 観測の積み上げによって、改善効果を**数字で語れる**ようになった（副産物としてアプリ開発との距離も縮まった）
5. **結論** — 数字で改善を語れる = **信頼性をエンジニアリングできる足場**ができた = それは紛れもなく SRE 活動だった

→ ここに直接寄与しない要素は、面白くても削る判断をする。

## 2. 全体ボリューム感の検討

`_notes.md` の初版は **6 セクション / 25 枚 / 30 分**。

メモ:

- 1 枚あたり平均 1 分強。30 分は意外と短い
- セクション 5「答え合わせ」が一番のハイライト → ここに時間を寄せたい
- セクション 2「NILTO 紹介」は 3 枚あるが、本筋はアーキテクチャ（Fastly → Cloud Run → Cloud SQL）の前提共有だけ。ここは圧縮余地あり
- 目標: **20 枚前後**まで絞り込む
- 現時点の per-section 仮見積（セクション 3 参照）: 4 + 2 + 3 + 4 + 4 + 4 = **21 枚**
- **当日の時間配分**: 25 分発表 + 5 分 Q&A（in-session）/ または 30 分発表（Q&A は別途受付サイト）。どちらでも対応できるよう、本体は 25〜28 分目安で組む

## 3. 各セクションの設計意図

### セクション 1: 導入・自己紹介（4 分）

- **聴衆に持って帰ってほしい感情**: 「自分も同じモヤモヤを抱えている」「この人の話なら聞ける」
- **避けたいこと**: 経歴自慢、SRE という肩書きを正面から名乗ること
- **判断済み**: 自己紹介は `index.md` 現行の two-cols のまま（変更しない）
- **スライド構成（仮 / 4 枚）**:
  1. タイトル
  2. 自己紹介（`index.md` 既存）
  3. おしながき（`index.md` 既存・本文要記入）
  4. インフラエンジニアの葛藤 + 「こんな人にこそ聞いてほしい」 —「SRE と呼べるのか?」の自問自答に、共感してほしい聴衆像を 1 枚に併記
- **おしながきの中身（仮）**: 6 セクション構成に合わせて以下で埋める案。`index.md` の現行テンプレは 7 項目あるが 6 に圧縮
  1. 自己紹介（このスライドの直前）
  2. NILTOについて
  3. 負荷試験、はじめました（Locust）
  4. 積み上げ式オブザーバビリティ
  5. 「答え合わせ」としての SRE
  6. まとめ
- **判断済み**: 「こんな人にこそ聞いてほしい」は別スライドにせず、葛藤スライド（4）に統合

### セクション 2: プロダクト紹介（NILTO）（4 分）

- **目的（2 つ）**:
  1. 後の負荷試験を理解するための最小前提（アーキテクチャ）
  2. **広報義務** — 出張費・工数を会社が負担しているので、NILTO の魅力を意図して伝える
- **負荷試験文脈で必須の 2 点**:
  - Fastly で CDN キャッシュしている
  - Cloud SQL がボトルネック候補になり得る三層構成
- **判断済み**: 2 枚構成。1 枚目を「NILTO カタログページ」として情報密度を上げる方針
- **スライド構成（2 枚）**:
  1. **NILTO の魅力ダイジェスト** — 以下の 4 機能を並べる「カタログページ」。各機能は名前 + 一行説明のみで踏み込まない（30 秒〜1 分で通過）
     - マルチサイト一元管理
     - メディアセキュリティ・監査ログ
     - フレキシブルテキスト
     - NILTO MCP（AI 連携）
  2. Developer API のアーキテクチャ — Fastly → Cloud Run → Cloud SQL の三層図 + 「Cloud SQL がボトルネック候補」
- **設計の勘所**:
  - 1 枚目は機能解説に時間を割かない。並べるだけで「こんなことができる」と印象づける
  - 1 機能 1 スライドの構成は避ける（時間を食うし、SRE 目当ての聴衆には冗長）
  - NILTO MCP は時流的に効くので入れたい

### セクション 3: 泥臭い第一歩（Locust）（6 分）

- **目的**: 「ないものは作る」精神 + 手軽さで聴衆の背中を押す
- **強み**: Locust の `@task` デコレータの簡潔なコードスニペットが効く
- **入り口の文脈整理（重要）**: 突発高負荷に対して**スケールアップという選択肢が取りにくかった**ので、限界性能を測りに行く必要があった、という流れにする。「場当たりスケールアップを反省して負荷試験へ」ではない
  - **公に言える理由**: Cloud SQL のスケールアップは**ダウンタイムを伴う**（インスタンス再起動）。サービス停止を伴う作業はおいそれと打てない
  - **オブラートで触れる**: インフラコスト効率を意識する文化がある（→ 「コスト面での慎重さ」程度に留める）
  - **言わない**: その他暗黙の社内事情。スライドにもノートにも書かない
- **スライド構成（3 枚）**:
  1. 事件発生 — Cloud SQL の突発高負荷。スケールアップはダウンタイムを伴うので即座には打ちにくく、まず**限界性能を知る**必要があった
  2. 「あるものでやる、ないものは作る」 — APM 不在の現実と Locust 採用。**環境変数 1 つで接続先（local / staging / CDN 経由）を切り替える Docker Compose の仕組みも自分で組んだ**ことを一行で添える（独立スライドにしない）
  3. Locust で小さく始める — `@task` のコード断片（1 画面に収まる量、3〜5 行）
- **判断済み**:
  - Docker Compose / `TARGET_ENV` は単独スライドにせず、スライド 2 に統合
  - スクリプト断片は 1 画面に収まる量（3〜5 行）に絞る

### セクション 4: 積み上げ式オブザーバビリティ（7 分）

- **目的**: 「完璧じゃなくていい」を体感してもらう
- **核**: Gemini 壁打ちのリアルさ。仕組みの完成度ではなく**プロセス**を見せる
- **冒頭で添える一言**: 「三本柱を順に揃えていく話ではなく、データを使って答え合わせができるようになるまでの積み上げの話です」
- **スライド構成（仮 / 4 枚）**:
  1. リクエストの「裏側」を知る必要性 — RPS だけでは DB の状態は分からない
  2. 自作テレメトリスクリプト — Cloud Monitoring API を直叩きして CPU・待機接続を抽出
  3. 過渡期のリアル — BQUploader はまだ未完成。「フルオートメーションされていなくていい」
  4. Gemini 壁打ち — 一人 SRE の相棒。インフラ指標とアプリ挙動の因果を解きほぐす
- **判断済み**: BQUploader 未完成スライド（3 枚目）は「自動化は途中で OK」の好例として確定して入れる

### セクション 5: 答え合わせ（7 分）★ハイライト

- **目的**: 負荷試験を「**問題発見ツール**」ではなく「**改善効果の検証ツール**」として捉え直す。これがオブザーバビリティ積み上げの最初の効能になる
- **対比軸の選択（重要）**: 「SRE = 犯人探し」という対比は世間一般に流布した誤解とは言えず、ストローマンになる。代わりに「**点で測るベンチマーク → 継続的な答え合わせ**」「**告発 → 検証**」の軸で語る
- **構造（崩したくない順番）**:
  1. CDN バイパスで 10 ユーザー / CPU 95% の特異点を発見（事実）
  2. コードに `FIXME 負荷検証と上限の制限` が残っていた（既知の負債）
  3. SRE が指摘する前にアプリ側がいつのまにか直していた
  4. 負荷試験データで改善効果を**後から定量化**できた（答え合わせ）
- **ビジュアル方針**:
  - 10 ユーザーで頭打ちになるグラフ → テスト結果レポートのスクショを**詳細マスクで掲載**
  - `FIXME` コメント → **コードスクショは出さない**。スライド上にテキストで「`FIXME 負荷検証と上限の制限`」と書く程度に留める
  - 「コードは見せないが、その存在は語る」という距離感がむしろ生々しさを出せる可能性
  - **Before/After グラフ（[images/before.png](./images/before.png) / [images/after.png](./images/after.png)）は 1 枚に並列配置**。Y 軸最大値が 30,000ms → 1,500ms（20 倍縮小）になっていることを自分で言及。callout で「20+ 秒 → 1 秒」を強調
  - **誠実さの確保**: 「キャッシュパージ時の特異な現象」と弁護せず、「**API に実際に重い処理があった。CDN がそれを隠していた**」と認める。Before のキャプションは「キャッシュ越しでは見えなかった、API の重い処理（95 パーセンタイル応答時間）」、After は「**同じ処理**が約 1 秒に」
- **スライド構成（4 枚）**:
  1. CDN バイパス試験で 10 ユーザー / CPU 95% の特異点（マスク済みグラフ）
  2. 原因 — 前後コンテンツ取得のために 10,000 件全件ロード（コードは出さずテキストで説明）
  3. `FIXME 負荷検証と上限の制限` のコメントが残っていた — 笑い所
  4. いつのまにか直っていた — Before/After を**1 枚に並列**配置（20 秒級スパイク → 1 秒）。負荷試験は告発ではなく**検証**の道具だった
- **判断済み**: 「ガラスの盾」（CDN キャッシュパージ時のリスク、Fastly Shielding 等）は本筋から外れるので削除。Q&A で聞かれたら話す
- **覚悟**: 他を削ってでもここに 7 分使う

  2. 数字で改善を語れる = **信頼性をエンジニアリングできる足場** = SRE の核心
  3. 「わたしがやっていることは、紛れもなく SRE だった」
  4. 今日からの第一歩 — スクリプト 1 本、API 1 つから / ご清聴ありがとうございました（Q&A 受付サイト案内も）
- **判断済み**: Q&A 用の想定問答スライドは置かない。代わりに**口頭対応用の想定問答メモ**を本ファイル末尾の `## 8. Q&A 想定問答メモ` で管理する

## 5. 「積み上げ式」とは何を積み上げたのか（語りの整理メモ）

タイトルの「積み上げ式オブザーバビリティ」は、聴衆によっては Observability の三本柱（Metrics / Logs / Traces）を順に揃える話と先入観で受け取られる可能性がある。本筋とズレるので、本文で誤解を予防する。

「積み上げた」と言える対象は何種類かあるが、本発表のストーリー（CFP 回収ライン: オブザーバビリティ↑ → 信頼性（SRE）との距離↓）に最も寄与するのは:

- **データ点の積み上げ** — 試験を繰り返すたびにメトリクスが蓄積され、改善効果を**後から数字で語れる**ようになった
- **「数字で語れる範囲」の積み上げ** — 数字で語れる = **信頼性をエンジニアリングできる足場**。CFP で約束した「信頼性（SRE）との距離が縮まる」の実体はここ
- **共通理解の積み上げ（副産物）** — 結果として開発と運用が同じデータを見る文化が育った。本筋ではないが副産物として価値あり

補助線として:

- 計測レイヤー（外形 → DB 状態 → アプリ挙動）の積み上げ ← ここを前面にすると三本柱と誤解されやすい。深追いしない
- 自動化度合い（手動 → スクリプト → API 収集 → BigQuery 集約）の積み上げ ← 「完璧じゃなくていい」の例として軽く触れる程度

**当日の語り方**: タイトルを補強する一言を発表中のどこか（導入か、セクション 4 の冒頭）で添える。例:

> 「三本柱を順に揃えていく話ではなく、データを使って改善効果を数字で語れるようになるまでの積み上げの話です」
