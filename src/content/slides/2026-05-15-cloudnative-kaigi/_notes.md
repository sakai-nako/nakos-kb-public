## **各セッションの詳細骨子と盛り込むべき知見**

### **1. 導入：インフラエンジニアのモヤモヤと事件の始まり**

* **インフラエンジニアの課題感:** IaC（Infrastructure as Code）やサーバー構築を日々こなしていても、プロダクトのビジネス価値や信頼性に直結している実感が湧きにくい。アプリケーションの内部構造が見えず、アプリ開発陣との間に見えない壁を感じている状態を共有する。  
* **きっかけ:** システムで発生した突発的なデータベース（Google Cloud SQL）の高負荷障害。「インフラのスケールアップだけで場当たり的に解決していいのか？」という疑問から、実際のシステムの限界性能を測るための負荷試験プロジェクトを立ち上げた経緯を語る。

### **2. 小さく始める負荷試験：「ないものは作る」精神**

* **ツールの選定:** エンタープライズ向けの立派なAPM（Application Performance Management）ツールがない環境でも諦めない。Pythonで書かれたオープンソースの負荷試験ツール「Locust」を採用し、自前で検証環境を構築した 1。  
* **手軽なスタート:** Docker (Podman) と Docker Compose を使い、環境変数（TARGET_ENV）一つでローカル環境やステージング、CDN経由など接続先を切り替えられる仕組みを作った 2。  
* **初級者へのティップス:** Locustのコードは非常にシンプル（HttpUserを継承し、ユーザーの行動シナリオを@taskとしてPython関数で書くだけ）で、初級者でもすぐに書き始められる。最初から本番環境の完全な再現を目指さず、まずは一番重要なAPI（例: コンテンツ取得API）を1つ叩くスクリプトから「小さく始める」ことの重要性を説く。

### **3. オブザーバビリティの「積み上げ」と過渡期のリアル**

* **自前計測スクリプトの作成:** Locustでリクエストを投げるだけでなく、システムの裏側の状態を知る必要がある。そこで、Google Cloud Monitoring APIを独自スクリプトで叩き、データベースのCPU使用率や待機コネクション数、可用メモリなどを抽出する仕組み（テレメトリ）を作り始めた 2。  
* **完璧じゃなくていいリアルな現状:** スクリプトで取得したメトリクスとLocustの実行結果を連携させ、BigQueryへ転送して一元管理する仕組み（BQUploader）などを設計しているが、現在も実装途上であり、完全な自動化には至っていない 2。初級者に向けて、「最初からダッシュボードがフルオートメーションされていなくてもいい。手動でのデータ突き合わせから少しずつ自動化を積み上げていけば十分な成果が出る」という生々しい現状を共有する。  
* **生成AIの活用:** 不完全なログやメトリクスの相関関係を読み解くために、生成AI（Geminiなど）を積極的に活用。インフラの指標とアプリの挙動の因果関係を解き明かすための「強力な壁打ち相手」として生成AIを利用したプロセスを紹介する。

### **4. 境界を越える：負荷試験は「暴く」ためではなく「答え合わせ」のために（分析結果の具体例）**

* **技術的負債の定量化:** CDN（Fastly）のキャッシュを意図的にバイパスして負荷をかけた結果、わずか「10ユーザー」の同時接続でCloud SQLが限界に達する特異点を確認した 1。この原因はアプリ側における「前後の記事を取得するために、一時的にDBの取得上限を10,000件に緩和して全件ロードする」という処理だった 2。  
* **「実は知ってました」というリアル:** 劇的なストーリーでは「インフラエンジニアが負荷試験でアプリの隠れたバグを暴いた！」となりがちだが、現実は異なる。ソースコードには FIXME 負荷検証と上限の制限 と記されており、アプリ開発側も設計段階から「ここが重い」という技術的負債をしっかりと認識していた 2。  
* **いつの間にか直っていた修正の「答え合わせ」:** 実際には、SRE側が「ここ重いですよ」と指摘する前に、アプリ開発側がいつの間にか該当箇所を修正してくれていたことも多かった。SRE側は、定期的な負荷試験のデータを後から比較することで、「いつのまにか直っている！」「あの修正で限界性能がここまで劇的に改善している！」ということを客観的なデータとして確認（証明）することができた。  
* **共通言語の獲得:** 負荷試験の真の価値は、誰かのミスを追及することではない。「やっぱり設計通りここが限界だったね」「直したらここまでスループットが良くなったね」と、インフラとアプリが同じデータを見ながら笑い合える「答え合わせ」のサイクルが生まれたことを共有する。

### **5. まとめ：オブザーバビリティを通じた「コンテキスト」の理解**

* システムの「コンテキスト」（インフラの設定、アプリケーションの認識している技術的負債、それらが絡み合った結果）をエンジニアリングし続けることの重要性。  
* 立派なツールがなくても、自前の負荷試験と泥臭いメトリクス収集から始めた活動が、結果的にシステムの透明性を高めた。それは「犯人探し」ではなく、開発と運用の「共通言語（データ）」を生み出す活動だった。  
* 「わたしがやっていることは、紛れもなくSREだった」。初級者も、今日から手元のスクリプト1つを書くことで、SREへの第一歩を確実に踏み出せるというメッセージで締めくくる。
