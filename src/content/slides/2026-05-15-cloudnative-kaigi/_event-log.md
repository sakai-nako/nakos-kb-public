#### GameDay

GameDay (2026-05-14) で 3 人チームとして参加した際の体験ベースの気付きメモ。
当日は時間制限がある中で複数の Quest を並行で潰していく形式だった。

- 3 人で Quest を手分けして、PR ベースで main にマージしていく流れは時間効率が良かった (PR #1〜#10)。
- ArgoCD で自動同期されるので、push したら勝手にクラスタに反映されるのは爽快感があった。
- 出題されている Quest はいずれも "現実によくあるやらかし" を模した内容で、原因の引き出しを持っているかが効いた。
  - イメージ名のタイポ (Quest 5: payment)
  - 設定値のタイポ (Quest 5: `valkley` → `valkey`)
  - probe のポートずれ (Quest 7: product-catalog の liveness probe)
  - メモリ不足での OOMKilled (Quest 7: checkout の memory limit 引き上げ)
  - アプリ側のキャッシュリークと意図的な `time.sleep` (Quest 9: recommendation)
  - Pod 可用性不足とロールアウト戦略不備 (Quest 10: ad の replicas=1 / 毎分 restart の CronJob)
  - NetworkPolicy / レート制限が想定外の通信まで巻き込む (egress-rate-shield)
- まず `kubectl get pods` で状態を確認 → `describe` でイベント → `logs` で詳細、という基本動作で半分くらいは原因が当たった。Grafana / Jaeger まで踏み込むのは「アプリ側が怪しい」「レイテンシ系」のときに有効だった。
- ArgoCDのダッシュボードが思った以上に便利だった
- "Image 名が違う" "ポートが違う" のような単純な凡ミス系は、答えがわかると一瞬だがエラーログが直接そう言ってくれるとは限らないので、まず Pod の状態 (ImagePullBackOff / CrashLoopBackOff / Running but not Ready) を切り分けるのが近道だった。
- デバッグ用に仕込まれた `time.sleep` がそのまま残っているパターンも現場あるあるで、ソースに目を通す価値を再認識した。
- セキュリティのために入れた "egress を絞る" 系のポリシーが、cluster 内部の Pod 間通信まで巻き込んでしまい、frontend で散発的に 5xx が出るという問題が興味深かった
    - 「DoS 踏み台化抑止」という良かれと思って入った baseline ポリシーが、`destination_cidrs: 10.0.0.0/8` を含んでいて Pod CIDR まで対象になっていた、というのは初見ではかなり気付きにくい。
- ad サービスの問題は単独だと「インプレッションが時々抜ける」程度の曖昧な症状で、原因は (1) replicas=1 (2) ad-deployer の CronJob が毎分動いて Pod を再起動 (3) graceful shutdown が無い、の合わせ技だった。
    - 対処は教科書通りで、`replicas: 2` / `maxUnavailable: 0` / `preStop sleep` / `terminationGracePeriodSeconds` / readiness と liveness probe の追加。落ち着いて 1 つずつ当てれば直る系。
- 「Pod は気軽に殺される前提で設計する」「単一 Pod に依存しない」という Kubernetes の基本原則を、実体験ベースで腹落ちできた。
- 簡単な Quest (タイポ修正、image 名修正) を最初にまとめて潰すと、PR を作って merge する流れに慣れることもできて精神的にも楽だった。
- ArgoCD の同期待ち時間に別の Quest を進める、というインターリーブができたチームの方が伸びやすい印象。
- "誰が何の Quest を触っているか" の同期方法を最初に決めておく (Issue / カンバン / 口頭でも何でも)と、さらに効率化できるかも

#### ユーザー体験を可視化する非同期処理の分散トレーシングとSLI計測：ジョブキューを跨ぐコンテキスト伝搬の仕組み

- ジョブキューを用いたシステムを舞台としたオブザーバビリティ
    - 非同期化・疎結合なシステム
    - 重い処理を負荷分散
    - リトライ可能にして耐障害性を高める
    - キューに貯めてバッチ処理することで効率化する
- 複数のキューを挟むと、一連の処理としてのつながりが見にくくなる
    - 複数のユーザーがいる場合に、さらに追いにくくなる
- コンテキスト伝播とスパンリンクで解消を目指す
- コンテキスト伝播
    - 例: 最初のサービスで採番したトレースIDを後続のサービスにHTTPリクエスト経由で引き回す
        - HTTPヘッダで伝搬
        - traceparentという値をヘッダに載せる
        - SQSだと使えないので、SQSのメッセージ属性に載せる
- トレース単位の設計
    - トレースを切る単位を工夫することで、見たい処理だけに集中して追うことができる

#### ペアーズ本番環境でのcgroup-aware化との死闘録

- HAProxyで、Aurora MySQLのリードレプリカをロードバランス
- 最大のターゲットは、Golangのガベージコレクタ
- 次に、Goランタイムスケジューラ（CPU使用率の15%）
    - 同時実行数は、ワーカースレッドの数だけ
        - GOMAXPROCSで設定
            - ただし、実際の並列実行はCPUコア数で頭打ち
- GOMAXPROCSを5にすると、レイテンシーが急増
    - CPUスロットリングが原因
        - 監視の死角だった
- 過剰並列->スケジューラのオーバーヘッド
- なぜスレッドを減らすだけでスロットリングが消えたのか
- CPU制限と過剰並列の衝突をほどく
- LinuxのCFSが、CPU時間を管理する
    - requests: cpu.weight
        - CPU時間の取り分に関わる、上限には関わらない
    - limits.cpu: cpu.max
        - ピリオド（期間）とクォータ（上限時間）で構成される
            - ピリオド内でのクォータを超えると、スロットリングされる
        - クォータはスレッド間で共有される
            - スレッドが多いほど、クォータの消費が早くなる
                - その分、スロットリングが発生しやすくなり、ピリオドの中で使えない時間が増えて、レイテンシーが増加
                    - 停止している時間が長いほど、平均CPU使用率のような集約値は実態とズレやすい
                        - 監視の死角の正体
- 教訓
    - 過剰並列を避ける
    - プロセスの並列数は、Requests/Limits/ワークロード特性を踏まえて決める
    - CPU使用率だけではなく、スロットリングも監視する
    - 異常検知まで含めたメトリクス・監視戦略

#### Terraformモジュールはなぜ「魔境」化するのか

- 魔境？: moduleの処理を読まないと何が作られるかわからない状態のこと
- DRY原則は、コードを重複させるな、ではなく、「知識を重複させるな」
- 処理と状態
    - 処理は、「誰が・何のために使うのか」というコンテキストに依存しない
    - 状態は、「誰が・何のために使うのか」というコンテキストに依存する
        - インフラは「状態」を書くので、コンテキストに応じてグループ化する（moduleにする）のがDRY
            - 「抽象化」をする
- 抽象化
    - 概念を定義することで、そのmoduleが何のためにあるのか決まる
        - では、その概念を誰のために定義するのか？
    - 何が必要で何が不要かの判断基準ができる
    - ラベリングされる - コードを読まなくても表す概念がわかる
- 利用するユーザーを考えた設計
    - ホワイトボックス・テンプレート（SRE・インフラ向け）
        - 「パーツ」だからこそ、シンプルにする（魔境化させない）
        - 「正しく書く」ための労力を、アーキテクチャ上の「意思決定」のみに集中
    - ブラックボックス・カプセル化（その他の人向け）
        - 利用者が「サービスを動かす」ことに集中できるようにする
        - 例えば、「APIサービス実行環境」そのもの
- 「共通化しない知識」を無理にまとめてしまうから、「魔境化」する
- 魔境化させては「いけない」のは、同じチームでモジュールの利用・管理までする場合
    - 逆に、魔境化していいのは、モジュールの利用と管理をやるチームが異なる場合

#### そのSLO99.9%、本当に必要ですか？   〜優先度付きSLOによる責任共有の設計思想〜

- リソースが足りない状況では、Devチームが自分たちのサービスの信頼性の向上に責任を持つようにする
    - SREは支援（Enabling）
    - Horizontal SRE
        - 「実際には何の業務も行わず、新たなゲート組織として認識される」リスク
- SREはサービスから離れすぎない、しかしDevも本番に責任を持つ
- Product-Focused Reliability for SRE
    - 信頼性の対象を、「サービス」から「プロダクトの重要機能」へ移すアプローチ
    - ビジネス観点からメトリクスを決めて、それがSLOになるイメージ
- SLOはいくつかの種類に分けられる
    - Service, Client-Side, End-To-End
- 責任をSLOで分けるという提案
    - 得意な領域で分けるのはアンチパターン
    - Service, Client-SideはDevで、End-To-EndはSRE(とPdMとか)、などがちょうどいいのかも
- End-to-end SLO: 機能の一連の流れで取れる指標をSLIとし、それを束ねてSLOとする
- 「99.9%は本当に必要か」はビジネスとの対話を起動するきっかけ
