- **セリフ**: 「きっかけは、ある日突然やってきた事件でした。Cloud SQL の CPU 使用率が、ほぼ天井に張り付いた。突発的な高負荷です。先ほどお話しした通り、Cloud SQL のスケールアップは **ダウンタイムを伴います**。ユーザーが触っている本番環境で、止まる時間を作るのは、かなり覚悟がいる判断です。コスト面でも、安易にインスタンスサイズを上げる、というのは打ちにくい。だったらどうするか — まず、**今のシステムの限界はどこにあるのか**、それを知っておかないと話が始まらない。これが、私が負荷試験に手を出すことになったきっかけでした。」

#### 3-2. あるものでやる、ないものは作る（duration 130s）

- **このスライドで**: 現場の制約と Locust 採用判断を見せる
- **なぜ**: 「ない」から手が止まる聴衆の背中を押す
- **何を伝えたいか**: 立派な APM がなくても、Locust と自前の Docker Compose で踏み出せる、という姿勢
- **セリフ**: 「ところが、いざ始めようとすると、足元には **エンタープライズ向けの APM がありませんでした**。商用ツールの導入を待っていると、いつまで経っても始められない。なので、**あるものでやって、ないものは作る** — これしかありませんでした。負荷試験のツールには、Python ベースの **Locust** を選びました。理由はシンプルで、Python なら自分が書ける、それと、**コードでシナリオを書ける** という軽さ。それから、`TARGET_ENV` という環境変数 1 つで、**ローカル / staging / CDN 経由** と接続先を切り替える Docker Compose を、自前で組みました。試験の対象が選べるって地味に大事で、後で出てくる『CDN バイパス試験』みたいなことも、この仕組みのおかげで素直にやれました。**立派な道具がないなら、揃ってから始めるんじゃなくて、足元の道具を組み合わせて始めればいい** — それが、このセクションでお伝えしたいことです。」
- **実装メモ**: APM 不在 / Locust 採用 / `TARGET_ENV` 1 つで接続先（local / staging / CDN 経由）切替の Docker Compose を自分で組んだ一行（独立スライドにせず統合）

#### 3-3. Locust で小さく始める（duration 120s）

- **このスライドで**: 具体物（コード）を 1 画面で見せる
- **なぜ**: 「これだけで始められる」と肌で感じてもらう
- **何を伝えたいか**: Locust の `@task` デコレータひとつで負荷試験は始められる、という軽さ
- **セリフ**: 「これは Locust で書いた、**ほぼ最小の負荷試験のコード**です。`HttpUser` を継承したクラスを作って、ユーザーが取る行動を `@task` というデコレータで関数として書く。それだけで、Locust がそれを並列で実行してくれて、応答時間や成功率などのレポートを出してくれます。たとえばこのコードだと、`/api/contents/...` というエンドポイントを叩く、というユーザーシナリオが 1 個。**1 関数 = 1 シナリオ** という対応関係が直感的で、シナリオを足したくなったら関数を増やせば終わりです。最初は試行錯誤しましたが、振り返ると、**最初の一歩はこのくらいの軽さで十分** だったな、と思っています。難しいツールを揃えるより、まず動くものを動かす — それが、負荷試験を続けるコツでした。」
- **実装メモ**: コード断片 3〜5 行（フォント 18pt 以上、1 画面に収める）

#### 4-1. リクエストの「裏側」を知る必要性（duration 80s）

- **このスライドで**: 自作テレメトリの動機を提示

#### 4-2. 自作テレメトリスクリプト（duration 120s）

- **このスライドで**: ツールではなく「仕組み」を見せる
- **なぜ**: APM がなくても観測できることを実体で示す
- **何を伝えたいか**: Cloud Monitoring API を直接叩けば、CPU・待機接続といったインフラ指標は自前で取れる、という事実
- **セリフ**: 「そこで作ったのが、**自作のテレメトリスクリプト** です。やっていることは単純で、Google Cloud Monitoring API を直接叩いて、Cloud SQL のメトリクスを必要な時刻範囲で抜き出す、というものです。取得しているのは、**Cloud SQL の CPU 使用率**、**待機コネクション数** — つまり接続待ちでブロックされているクエリの数ですね、それから可用メモリと、ディスクの I/O。これらを、Locust を実行した時刻と突き合わせて、CSV に並べます。後はスプレッドシートで眺める、という地味な工程です。やっていることは単純ですが、**APM の代わりが、API ひとつで自分の手元に揃った** というのが、けっこう大きな手応えでした。立派な観測スタックがなくても、**観測の仕組みは作れる** ということを、自分の手で確認した瞬間です。」
- **実装メモ**: 構成図 or 短いコード（CPU・待機接続の抽出を抜粋）

- **なぜ**: 完璧主義からの解放。「フルオートメーションされていなくていい」を腹落ちさせる

- **セリフ**: 「ただし、正直に言うと、この観測の仕組みは **まだ完成していません**。本当は、テレメトリスクリプトの結果を BigQuery に自動で送って、ダッシュボードでまとめて見られるようにしたい — そう思って **BQUploader** という仕組みを設計したんですが、ここはまだ未実装。**ダッシュボードもまだ作っていません**。今は CSV に出力したものを、スプレッドシートで眺めて、Locust の結果と目視で突き合わせる、という、けっこう手作業の世界です。それでも、**この手作業の突き合わせから、すでに価値は出ている** んです。完成度ではなく、**今ある不完全な仕組みで、何を語れるか** — それが大事だと思っています。」
- **実装メモ**: 現状の手動部分 / 未完成の積み残し / 「途中で OK」の一言

#### 4-4. 生成 AI（Gemini）との壁打ち（duration 120s）

- **このスライドで**: 一人 SRE の相棒として AI の使い方を共有
- **なぜ**: 仕組みの完成度ではなく**日々の行動**そのものを魅力にする
- **何を伝えたいか**: 生成 AI は仮説を一緒に作る相棒として機能する、というリアルな使い方
- **セリフ**: 「一人で SRE 的なことをやっていると、考えを整理する相手が欲しくなります。一緒に首をひねってくれる相棒が欲しい。私の場合、その相棒は **Gemini** です。具体的にどう使っているかというと、たとえば、CSV にまとめた Cloud SQL の CPU 推移と Locust の応答時間グラフを並べて、『この同じ時間帯に CPU が跳ねているのに、応答時間がそんなに悪化してないのは、どう解釈する?』と投げる。すると、Gemini はキャッシュ層の影響、コネクションプールの飽和タイミング、いろんな仮説を一緒に出してくれます。それらを叩き台にして、私が次の試験設計を考える — そういう壁打ちです。**不完全なデータでも、相棒がいれば、語れる**。AI が答えを出してくれるんじゃなくて、**仮説を一緒に作ってくれる存在** として、想像以上に機能してくれてますね。」
- **実装メモ**: 投げている問いの例 1〜2 個（インフラ指標とアプリ挙動の因果を解きほぐす対話）

### 5. 「答え合わせ」としての SRE

#### 5-0. セクション区切り「答え合わせとしての SRE」（duration 10s）

- **このスライドで**: ハイライトへの導入
- **なぜ**: 物語の山場であることを場の空気で示し、聴衆の期待を高める
- **何を伝えたいか**: ここから負荷試験を「告発」から「検証」へ転換させる、という予告
- **セリフ**: 「ここからが今日の山場です。負荷試験の結果が、ある『**答え合わせ**』として返ってきた話をします。」

#### 5-1. CDN バイパスで見えた特異点（duration 100s）

- **このスライドで**: 衝撃的な事実を提示
- **なぜ**: 数字で隠れていた問題のスケール感を体感してもらう。物語のフックを作る
- **何を伝えたいか**: わずか 10 ユーザーで Cloud SQL の CPU が 95% に張り付くという事実。CDN に隠れていたボトルネックの存在
- **セリフ**: 「ある時、私は **CDN をあえてバイパスして**、Cloud Run の API を直接叩く負荷試験をかけてみました。普段ならエッジで Fastly に吸われるリクエストを、奥の API に全部届ける、という試験設定です。結果がこれです。**わずか 10 ユーザーの同時接続で、Cloud SQL の CPU が 95% に張り付きました**。応答時間も急激に悪化しました。たった 10 ユーザーですよ。**CDN というキャッシュ越しでは、まったく見えなかった特異点が、剥き出しで現れた瞬間** でした。CDN は普段は素晴らしく仕事をしてくれますが、その一方で、**裏側のボトルネックを覆い隠してしまう** という性質も持っている。これが、後の物語のフックになります。」
- **実装メモ**: マスク済みのテスト結果スクショ + 短い注釈（CDN バイパス試験という条件）

#### 5-2. コードに潜む「1 万件全件ロード」（duration 100s）

- **このスライドで**: 原因を語る（コードは出さない）
- **なぜ**: コードを出さずテキストで語ることで、生々しさと聴衆の想像力を引き出す
- **何を伝えたいか**: 特異点の正体は API の中にいた重い処理（前後コンテンツ取得のために 10,000 件全件ロード）だった、という事実
- **セリフ**: 「では、その特異点の正体は何だったか。**原因はコードの中にいました**。あるエンドポイントが、**ある条件下では**、前後のコンテンツを取得するために、DB の取得上限を一時的に **1 万件まで緩和** して、**そのフルレコードをまるごとメモリにロード** していたんです。レコード 1 件 1 件は決して大きくありません。でも、1 万件分のデータをメモリに展開するとなると、これは普通に重い処理です。**そりゃ、CPU が天井に張り付くわけですよ**。コードそのものは、あえてお見せしません。皆さんも、自分のところのコードベースを思い浮かべて、**こういう処理、心当たりありませんか?** その想像力を、ぜひこの先のお話に持ち込んでください。」
- **実装メモ**: 説明テキスト + 簡単な図解。コード自体は出さない

#### 5-3. `FIXME 負荷検証と上限の制限`（duration 110s）

- **このスライドで**: アプリ側の自覚（FIXME コメント）を提示
- **なぜ**: 笑い所。「ね、わかってたんですよ」で場を温め、SRE = 告発者という構図を崩す
- **何を伝えたいか**: アプリ側もこの問題を自覚していた、という事実。SRE と開発が同じ問題を見ていた
- **セリフ**: 「で、ここからが面白いところです。その重い処理の、すぐ近くに、こんなコメントが残っていました。…… **`// FIXME 負荷検証と上限の制限`**。…… **ね、わかってたんですよ**。アプリ側も、ちゃんと、わかってたんです。『負荷検証』と『上限の制限』 — つまり、『この上限値、本当に大丈夫?』『どこかでちゃんと検証して、上限制限を入れないとマズいよね』という認識が、コードのコメントとして残されていた。SRE が後から発掘したわけじゃなくて、**書いた本人が、書いた瞬間から自覚していた** ということです。私はこれを見つけたとき、**笑っちゃいました**。SRE と開発のあいだに溝があって、SRE が一方的に問題を告発する、みたいな構図ってたまに語られますが、現実はそうじゃない。**同じ問題を、同じ時に、同じように見ていた** — その物的証拠が、このコメントだったわけです。」
- **実装メモ**: FIXME 文字列の引用 + 笑い所演出のレイアウト（間を取れる余白）

#### 5-4. いつのまにか直っていた（duration 100s、最重要）

- **このスライドで**: Before/After で改善効果を可視化
- **なぜ**: 物語のクライマックス。本発表の核を体感してもらう
- **何を伝えたいか**: 改善効果は数字で語れる（**20+ 秒 → 数秒**）。負荷試験は告発ではなく **検証** の道具だった
- **セリフ**: 「そして、ある時、**いつのまにかこれが直っていました**。…… 正確には、4 月末にチームで修正がリリースされていた、というのが正しいんですが、私の体感としては『気がついたら直っていた』に近いです。Before / After を見てください。Before では応答時間が **20 秒、30 秒のオーダー**。After では、Y 軸そのものが大きく縮小して、**数秒のスケール** に収まっています。ここが面白いところです。**直したのは、FIXME のあった場所じゃないんです**。**フルレコード取得を、ID のみの取得に変えた、それだけ**。**1 万件取得自体は、今も変わっていません**。根本に手を入れなくても、効くところに効かせれば、**信頼性は engineering できる**。負荷試験は **告発の道具** じゃなくて、**検証の道具** だった、という、私にとっての答えでした。」
- **実装メモ**:
  - Before / After グラフを左右並列配置
  - 中央に「**20+ 秒 → 数秒**」の callout
  - キャプション:「キャッシュ越しでは見えなかった、API の重い処理（95 パーセンタイル応答時間）」「**同じ処理**が数秒に」
  - Y 軸最大値が大きく縮小していることに**自分で言及する**注釈
  - **「直したのは FIXME 箇所ではなく、フルレコード取得 → ID のみ取得への変更」「1 万件取得自体は据え置き」**はセリフで補足する（スライド本体には書かない）
