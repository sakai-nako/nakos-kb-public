# 媒体別アダプタ — ブログ記事

[00-core.md](./00-core.md) の観点を、ブログ記事の構成に翻訳します。コアを置き換えるものではなく、ブログ固有の勘所を足すものです。記事の置き場は [src/content/blog/](../../src/content/blog/)（現状はプレースホルダーのみで、コレクション定義・公開ページは未実装。下書きは [content-external/articles/](../../content-external/articles/) にも置ける）。

---

## 不変条件のブログ版（均衡モデル）

| 段階           | 中身                                       |
| :------------- | :----------------------------------------- |
| 均衡（現状）   | 読者がいる前提・状況。共有できる出発点      |
| 不均衡（課題） | 困り事・疑問・引っかかり。記事が扱う問題    |
| 新たな均衡     | 解決後の世界・新しい見方。読後に残る変化    |

→ 記事を書く前に「読み終えた読者は**何ができる／何を感じる**ようになるか」を一文にする（[00-core.md](./00-core.md) 観点 2）。

## ブログ固有の勘所

### 冒頭で「困り事の一行代弁」をする

読者の困りを 1 行で言い当てる。これがフックになる。

- 根拠: [playbook.md](../../docs/learnings/playbook.md) 原則 2。例:「○○最近聞くけど、どのツールを使えばいいかわからない」。
- ❌ 「○○について書きます」と宣言から入る → 引きが弱い。

### 結論先出し（プロレプシス）を基本に

Web 記事は途中離脱が前提。**先に結論・要点を出してから**理由・例で支える。

- 型: **PREP**（結論 → 理由 → 例 → 結論）/ **SDS**（要約 → 詳細 → 要約）。
- 物語の引きを作りたい記事（体験談・ポエム）はあえて伏線型（アナレプシス）にしてもよいが、技術解説は結論先出しが安全。

### 見出しだけで筋が通るように設計する

`##` 見出しを上から読むだけで「均衡 → 不均衡 → 新たな均衡」の弧が見えるか。見出しは**目次でありプロット**。

- ❌ 「はじめに／本題／まとめ」のような無情報見出し。
- ⭕ 各見出しが「何が起きたか」を語っている。

### 焦点化（語りの立場）を決める

体験談なら一人称（内的焦点化）、技術解説なら俯瞰（ゼロ焦点化）。主語が「私／我々／あなた」で揺れないようにする（[00-core.md](./00-core.md) 観点 5）。

### アクタントで素材を整理する

読者＝主体、解決したい課題＝対象、立ちはだかる問題＝敵対者、紹介するツール／知見＝助力者。この配置が決まると「何を説明すべきか」が自ずと出る。

## frontmatter との接続

`blog` の frontmatter（`title` / `description` / `pubDate`）は構成の要約でもある。

- `title`: **幹**（[00-core.md](./00-core.md) 観点 1）を読者の言葉に直したもの。困り事の代弁を兼ねると強い。
- `description`: 「誰に・何を残すか」（観点 2）を 1〜2 文で。検索結果・カードに出る最初のフック。
