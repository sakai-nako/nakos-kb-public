# 認知物語論 — 受け取られる側のモデル

[01-narratology.md](./01-narratology.md) が「**どう語るか**（生成・構造）」の語彙なら、こちらは「**どう受け取られるか**（受容・認知）」の語彙です。[00-core.md](./00-core.md) の (A)-3 物語性 と 観点 6（具体観察 → 感情ラベル）の理論的裏打ちにあたります。

> 構成は「書いて終わり」ではなく、受け手の脳内で意味が立ち上がって初めて成立します。このファイルは「**書いたもの（言説）と、立ち上がる体験（物語性）は別物だ**」という当たり前を、設計の道具に変えるためのものです。

---

## 1. 二つの軸 — 生成スタックと受容の対

物語内容・物語言説・物語性（narrativity）は、一直線の段階ではなく **性質の異なる 2 つの軸** で捉えると設計に効きます。

- **軸 A：垂直の生成スタック** — 物語内容（素材）から物語言説（提示）への投影。ソフトウェア設計の **Model → View** に対応。[00-core.md](./00-core.md) (B) の「素材と語りの分離」がこの軸。
- **軸 B：生成物 ⇄ 受容体験** — 物語言説（書いたもの）と物語性（読み手の中に立つ反応）の対。これは **UI ⇄ UX** とまったく同じ関係。

| 物語論 | ソフトウェア | 性質 |
| :----- | :----------- | :--- |
| 物語内容（fabula） | ドメインモデル / データ | 表現に依存しない素材 |
| 物語言説（récit） | UI / View | 設計者が直接書けるアーティファクト |
| 物語性（narrativity） | UX | **直接置けない。言説を通じて誘発するしかない受容効果** |

→ ここがいちばん重要: **設計者が手で触れるのは物語言説（UI）だけ**。物語性（UX）は「狙って配置」できず、言説の設計を通じて間接的に「誘発」するしかない。「面白くならない」と詰まったとき、いじれるのは常に言説の側だと自覚する。

---

## 2. ノーマンの三モデルとの対応

ドナルド・ノーマン『誰のためのデザイン？』の 3 モデルは、物語論の 3 層にそのまま重なります。これが「構成＝インターフェース設計」と捉える根拠です。

| ノーマン | 物語論 | 中身 |
| :------- | :----- | :--- |
| デザインモデル（Design Model） | 物語内容（fabula） | 作り手の頭の中の構想 |
| システムイメージ（System Image） | 物語言説（récit） | 実際に作られた人工物＝テクスト |
| ユーザーのメンタルモデル（User's Model） | 物語性（narrativity） | 受け手が人工物から形成する理解 |

認知物語論でいう「**センスメイキング**（受け手が脳内で意味を組み立てる過程）」は、このメンタルモデル形成そのものです。

→ 制御できるのは中央のシステムイメージ＝言説だけ。両端のモデル（作り手の構想 / 受け手の理解）は内面にあって直接触れない。**良い構成とは、言説を介して両端を一致させる営み**。「自分が意図した幹（観点 1）が、読後に読者の中で同じ幹として立つか」を問うのはこの一致の検証。

## 3. 欠落のデザイン — 推論を誘発する空白

受け手は情報の穴を自分で埋めようとします。**あえて語らないことで推論を起こさせる**のが、物語性を最も効率よく立ち上げる手口です。

- 典型: ヘミングウェイの 6 語「For sale: baby shoes, never worn（売ります、赤ん坊の靴、未使用）」。構造はほぼ無いのに強い物語性が立つ。穴を読み手が埋めるから。
- 物語論の名前: **黙説法（paralipsis）** — 語るべき情報をあえて伏せる変調。

| 媒体 | 欠落のデザイン例 |
| :--- | :--------------- |
| ブログ | 結論を全部書かず、最後に読者自身の問いとして残す |
| 発表資料 | 答えを先に言い切らず、「答え合わせ」まで引っ張る（[20-slides.md](./20-slides.md)） |
| 小説 | 心情を説明せず、行動や情景だけ描いて読者に推し量らせる（[30-novel.md](./30-novel.md)） |

→ [01-narratology.md](./01-narratology.md) の省略法（ellipsis）・要約法（summary）と地続き。[00-core.md](./00-core.md) 観点 7（意図的に削る）の「削った先に何が立つか」を説明する観点でもある。

❌ **欠落が「説明不足」に転ぶ線引き**: 推論の手がかり（埋めるための材料）まで消すと、ただの不親切になる。「穴は空けるが、埋め方は示す」。

## 4. SSOT からの多媒体投影

同一の物語内容（Single Source of Truth）を保持したまま、媒体ごとに言説を変調できる——これは軸 A の直接の帰結です。

- 同じ体験（fabula）から、ブログには親しみやすい語り、発表資料には要約法で締めた構成、小説には情景法の描写を、幹を保ったまま投影する。
- これは **このフレームワーク自身の構造**でもある: 共通コア（[00-core.md](./00-core.md)）＝物語内容、媒体別アダプタ（[10-blog.md](./10-blog.md) / [20-slides.md](./20-slides.md) / [30-novel.md](./30-novel.md)）＝媒体ごとの言説生成器。

→ [00-core.md](./00-core.md) (B)「素材と語りの分離」が「なぜ自由に設計できるのか」の裏付け。素材は 1 つ、言説は媒体の数だけある。

## 5. アナロジーの臨界点（誠実な注記）

軸 A を「Model → View」と言い切るのは便利ですが、物語では成り立たない場面があります。

- ソフトのドメインモデルは「正しさ・表現非依存」を一次的価値として **独立に** 存在できる。
- だが物語の物語内容（fabula）は、構造主義的には **受け手が言説から事後的に再構成する抽象** で、言説に先立って独立に在るとは限らない。**しばしば言説が内容に先行する**。

→ 実務的含意: [00-core.md](./00-core.md) (B)-1「素材を時系列で全部書き出す」で fabula が固まりきらなくても問題ない。**書きながら（言説を試しながら）素材の方が立ち上がってくる**のは正常。fabula → syuzhet は一方通行ではなく双方向の往復だ、と捉える。
