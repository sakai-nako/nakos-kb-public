# Novel Revision Workflow（小説推敲フロー）

このスキルは [src/content/novels/](../../../src/content/novels/) 配下の自作小説原稿を、対話的かつ段階的に推敲するためのワークフローです。

前提として [.claude/rules/35-novels.md](../../rules/35-novels.md) の記法・配置ルールと [src/content/novels/README.md](../../../src/content/novels/README.md) のスタイルに従ってください。本スキルはそれらを置き換えるものではなく、推敲というタイミング限定の作業フローを規定します。

## 起動条件

ユーザーが novels 原稿の見直し作業を始める意思を示したとき:

- 「(全体的に / 通しで / ざっと) 推敲したい」「推敲してほしい」「推敲を手伝って」
- 「読み返したい」「読み直したい」
- 「文章を整える / ブラッシュアップ / 仕上げたい」
- 「(この章 / このエピソード) を見直したい」
- 章ファイルを開いた状態で「だいたいできた」「一通り書けた」など完成を匂わせる発言

対象は **原則 1 章ずつ**。複数章を一度に対象にしたい場合はユーザーに明示的に確認する。

## 全体方針（テンポ優先）

ユーザーは趣味で書いており、テンポを最優先する。具体的には:

- **論点を 2-3 個に絞って提示する** — 一度に 5 つも 6 つも出さない
- **ユーザーが選択した論点だけを修正する** — 全部直そうとしない
- **修正は最小単位** — 1 文〜数文の編集に留め、章全体の再構成のような大きい書き換えは原則しない
- **大きい論点（テーマ・設定・プロット）には踏み込まない** — このスキルは文章レベルの推敲が対象
- **セリフの口調や方言の質感を尊重する** — 関西弁の細部や口癖は作者の意図的な選択

このリズムが崩れる動きはやらない:
- ❌ 論点を一気に 6 つ出して「どうですか」
- ❌ ユーザーが選んでない論点を「ついでに」直す
- ❌ 章全体を書き換えた diff を提示する
- ❌ 「全部直しておきました」と事後報告する

## 推敲フェーズ

### フェーズ 0: 初期スキャン

ユーザーが推敲対象を明示していない場合、まず確認する:

- IDE で開かれている章ファイル（`<ide_opened_file>` から取得可能）
- 直近で編集された章（`git status` / `git log`）

対象が決まったら:

1. 章本文を **Read** で読む
2. 同じ作品内の前後章があれば **軽く眺めて連結を意識**（前章で立てた伏線、次章での回収予定）
3. 作品 `index.md` を読み、synopsis や status を把握

このフェーズで余計な情報を取りすぎないこと。**前後章は推敲対象ではない**ので深く読まない。

### フェーズ 1: 観点の選択

以下 3 観点を提示し、どれから始めるか選んでもらう。順番は通常 1 → 2 → 3 が自然:

1. **モチーフ・流れ** — テーマや象徴の連鎖が読者に伝わるか、テンポが詰まりすぎ／緩みすぎていないか
2. **方言・セリフ** — 各キャラクターの口調の差別化、関西弁と標準語の対比、セリフの自然さ
3. **心情描写の密度** — 主人公の「気づき」のスローダウンが説明的になりすぎていないか、体感に寄せられているか

「全部やる」と言われたら 1 → 2 → 3 を順に。「ここだけ」と特定の観点を指す場合はその観点に集中。

### フェーズ 2: 各観点の通し読み

各観点について以下のループを回す:

1. **章を Read** で再読（必要なら前後章の関連箇所も）
2. **論点を 2-3 個に絞って提示**
3. **ユーザーが選択** — 全部 / 一部 / 全部スキップ
4. **選ばれた論点を 1 つずつ Edit で修正** — 最小単位の編集
5. **次の観点へ**

論点提示の定型フォーマット:

```
### 観点 N で見つけた論点

**(α) line XX の◯◯**
[何が問題か / 気になるか — 該当箇所を引用しつつ簡潔に]
→ [修正の方向性 — 具体例つき。1 案でも複数案でも可]

**(β) line YY の△△**
...

---

[2-3 個の論点をどう判断するか短く補足。例: 「α は機械的に直せる、β は意図次第」]

どれから手を付けますか？
```

論点ラベルは a, b, c, ... と続け、フェーズをまたいでも振り直さず通し番号で運用すると、ユーザーがあとで参照しやすい（例: a, b, c → d, e, f → g, h, i → ...）。

### フェーズ 3: 通し読み（細部チェック）

3 観点を一通り終えたら、最後に頭から 1 周ナメる。以下のチェック対象を順次:

#### 3-1. 表記揺れ

`Grep` を使って候補語の頻度を出し、揺れがある箇所を特定:

- 漢字 vs ひらがな: `少し / すこし`、`時 / とき`、`一人 / ひとり`、`事 / こと`、`物 / もの`、`所 / ところ`、`為 / ため` など
- 漢数字 vs アラビア数字（**基本は漢数字で統一**）
- 旧字体 vs 新字体: `眼 / 目` など

セリフ内の表記揺れは **キャラクター個性として残しても良い**ことが多いので、地の文と分けて報告する。

#### 3-2. 視点の移り変わり

- 一貫した limited third-person か（誰の視点で描かれているか）
- 主語省略で動作主が一瞬わかりにくい箇所がないか
- 別キャラクターの主観に滑り込んでいないか

#### 3-3. 繰り返し語

`Grep` で頻度を数える。短い区間で密集していたら指摘:

- `軽く+動詞` `そっと+動詞` などの常套句
- `ぼんやり`「漠然と」「なんとなく」「ふと」など認知の朧げさを示す副詞
- 主語名（主人公名）の同一段落内での連続反復
- 場所・小道具（バー設定なら `グラス` など）の二重指示や同一文中の反復

頻度の絶対値より「**短い区間で密集しているか**」を見る。バー場面で `グラス` が 17 回出ても、間隔があれば問題ない。

#### 3-4. 小説特有のチェック（記法）

[35-novels.md](../../rules/35-novels.md) の記法ルールに照らして:

- ルビ表記: `|漢字《かな》` の半角 `|` + 全角 `《》`、混在していないか
- 傍点表記: `《《傍点》》` の整合性
- 段落頭の字下げ: 全角スペース 1 つ
- 章タイトルが frontmatter にあり、本文で `#` 見出しを使っていないか

細部チェックでも、**論点を 2-3 個に絞って提示** → ユーザーが選ぶ → 修正、のループを維持。

### フェーズ 4: メタ更新

修正が完了したら作品の `index.md` の `updated_at` を今日の日付に更新（[35-novels.md](../../rules/35-novels.md) のルール）。今日の日付は `currentDate` から取得。

## 終了条件

以下のいずれか:

- ユーザーが「もういい」「これで完成」「これで OK」と宣言
- すべての観点と細部チェックで「これ以上ない」状態
- ユーザーから別タスクへの移行サインが出る

終了時は **最終差分のサマリー** を出して終了:

```
### 最終差分まとめ

**観点1（モチーフ・流れ）:**
- (b) line XX を ... に圧縮
- (c) line YY に ... を差し込み

**観点2（方言・セリフ）:**
- (d) line ZZ ... → ...

...

**観点3（心情描写の密度）:**
- ...

**通し読み:**
- 表記揺れ: ...
- 繰り返し語: ...

`updated_at` も今日 (YYYY-MM-DD) に更新済み。
```

## 振る舞いのコツ

### やってはいけないこと

- **章全体の書き直しを提案しない** — 文章レベルの推敲が対象。大規模な再構成は別タスクとして切り出す
- **記法を変えるような提案をしない** — `#` 見出しを増やす、frontmatter スキーマを増やす、ルビ記法を変える等は novels/README.md のルール違反
- **論点を 5 つも 6 つも一度に出さない**
- **ユーザーが選んでない論点を勝手に直さない**（「ついでに直しておきました」は避ける）

### やったほうがいいこと

- **修正の差分を簡潔にサマライズする** — `(α) line XX を ◯◯ に圧縮` のような形式
- **連作中なら前後章との接続を意識する** — 「次章で繋がるからここはこのまま残す」など、ユーザーが連結を考えて判断できる材料を提示
- **ユーザーが手動で直した変更を尊重する** — `system-reminder` で intentional な変更が示された場合、それを上書きしない
- **論点を提示する時、トレードオフを 1 行添える** — 「α は機械的に直せる、β は意図次第」のような判断材料
- **修正提案は具体例つきで** — `→ 例: 〜` の形で代替案を示す
- **複数の Edit を並列実行できる時は並列で** — 互いに干渉しない位置の修正は 1 メッセージにまとめて投げる
