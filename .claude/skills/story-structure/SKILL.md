# Story Structure Workflow（ストーリー構成フロー）

ブログ・発表資料・小説に共通する「構成（ストーリーの組み立て）」を、対話的かつ段階的に設計・レビューするためのワークフローです。

前提として [.claude/storytelling/](../../storytelling/) を SSOT として参照します（共通コア [00-core.md](../../storytelling/00-core.md)、物語論辞書 [01-narratology.md](../../storytelling/01-narratology.md)、媒体別アダプタ [10-blog](../../storytelling/10-blog.md) / [20-slides](../../storytelling/20-slides.md) / [30-novel](../../storytelling/30-novel.md)）。本スキルはそれらを置き換えず、構成を組み立てる／診断するという作業フローを規定します。

## 起動条件

- 「構成を考えたい」「組み立てたい」「アウトライン／目次／章立てを作りたい」
- 「話の流れ／順番を整理したい」「導入が弱い気がする」「オチが決まらない」
- 「この記事 / スライド / プロットの構成をレビューして」「ここ、伝わる？」
- ブログ・スライド・小説のいずれかで、文章を書く前の**設計**、または書いた後の**構成診断**を求める発言

> 文章そのものの磨き込み（語彙・リズム・表記）は構成ではない。小説の推敲は [novel-revision Skill](../novel-revision/SKILL.md)、と切り分ける。迷ったらユーザーに「構成（流れ）の話か、文章の話か」を確認する。

## 全体方針（テンポ優先）

ユーザーはテンポを重視する。novel-revision と同じ流儀で進める。

- **論点を 2-3 個に絞って提示する** — 一度に 5 つも 6 つも出さない
- **ユーザーが選んだ論点だけを進める** — 全観点を機械的に埋めようとしない
- **観点は名指しする** — [00-core.md](../../storytelling/00-core.md) の観点番号や物語論の用語（[01-narratology.md](../../storytelling/01-narratology.md)）で「いま何の話をしているか」を共有する
- **完成形を押し付けない** — 型は道具。ユーザーの狙いが最優先

このリズムが崩れる動きはやらない:
- ❌ 8 観点を一気に全部チェックして「どうですか」
- ❌ ユーザーが選んでいない観点まで勝手に作り込む
- ❌ 物語論の用語を、転用例なしで振りかざす

## フェーズ

通常は 0 → 6 を順に。途中から（例: 「導入だけ見て」）入る場合は該当フェーズに直行する。

### フェーズ 0: 媒体と狙いの確定

- **媒体**（ブログ / スライド / 小説）を確認し、対応するアダプタ（[10](../../storytelling/10-blog.md) / [20](../../storytelling/20-slides.md) / [30](../../storytelling/30-novel.md)）を頭に入れる
- **狙い**: 何のために書くか（バズ / 深く刺す / 記録 / 採択 など）。[90-principles.md](../../storytelling/90-principles.md) 原則 5 の「拡散 vs 深さ」で軸を確認
- レビュー依頼なら、対象（構成案 / 既存原稿）を **Read** で把握

### フェーズ 1: 幹を一言にする

[00-core.md](../../storytelling/00-core.md) 観点 1。「これが抜けたら崩れる」一文を一緒に決める。

- 幹が 2 つあるなら、絞るか上位の幹を立て直す
- 出てこないなら、ニッチ趣味との接続（[90-principles.md](../../storytelling/90-principles.md) 原則 3）から手繰る

### フェーズ 2: 対象と持ち帰らせたい変化

観点 2。**誰に / 何を残すか / 避けたいこと**を言葉にする。アクタント（読者＝主体、課題＝対象…）で素材の役割を整理する。

### フェーズ 3: 素材出し（fabula）→ 並べ替え（syuzhet）

[00-core.md](../../storytelling/00-core.md) の (B)。構成作業の核。

1. **素材を時系列で書き出す**（順番・面白さは考えない）
2. **聞き手に効く順番へ並べ替える**（結論先出し＝プロレプシス / 伏線・回想＝アナレプシス）

ここで「何を書くか不足（fabula）」なのか「並べ方の問題（syuzhet）」なのかを切り分ける。

### フェーズ 4: 構成の型を選ぶ

観点 3。狙いと媒体に合う型を 1 つ。起承転結 / 三幕 / 問い→過程→答え / PREP / SDS。型は均衡モデル（[01-narratology.md](../../storytelling/01-narratology.md)）の変奏だと意識する。

### フェーズ 5: 語りのコントロールと配分

観点 4・5。ジュネットの 3 軸で緩急を設計する。

- **順序**: 結論先出しか、伏線型か
- **速度**: ハイライトに分量／時間を寄せ、周辺は要約・省略
- **焦点化**: 誰の視点で語るか、一貫しているか

### フェーズ 6: 削るものの仕分け

観点 7。幹に寄与しない要素を、面白くても削る。削ったネタは捨てず Q&A・脚注・続編へ退避先を決める。

### レビューモードのとき

既存の構成案／原稿を診断する場合は、フェーズを「作る」から「点検する」に読み替える。[00-core.md](../../storytelling/00-core.md) と [01-narratology.md](../../storytelling/01-narratology.md) の観点で、論点を 2-3 個ずつ提示:

```
### 構成レビュー: 見つけた論点

**(a) 幹がぼやけている（観点 1）**
[引用 or 該当箇所] — 「A の話」と「B の話」が同居している。
→ どちらかに絞るか、上位の幹「○○」に束ねる案。

**(b) ハイライトの配分（観点 4 / 速度）**
[該当箇所] — 山場が要約で流れている。ここを情景法で厚くしたい。

---
a は方向性の判断、b は機械的に直せる。どれから見ますか？
```

論点ラベルは a, b, c… と通し番号で運用すると後で参照しやすい。

## 重い作業の委任

ゼロからの構成ドラフト生成や、長い原稿一式の網羅的レビューは、媒体別 SubAgent に投げて結果を受け取れる:

- ブログ: [blog-architect](../../agents/blog-architect.md)
- スライド: [slide-architect](../../agents/slide-architect.md)
- 小説: [novel-architect](../../agents/novel-architect.md)

Skill（対話・小出し）と SubAgent（投げ込み・まとめて返す）の住み分けは [rules/36-storytelling.md](../../rules/36-storytelling.md) を参照。ユーザーと一緒に練りたいときは Skill のまま、丸ごと任せたいときは SubAgent を提案する。

## 終了条件

- ユーザーが「これでいい」「書き始める」と宣言
- 構成が一通り固まり、次は執筆フェーズに移る
- レビューで「これ以上ない」状態

終了時は**決まった構成のサマリー**を出す:

```
### 構成まとめ（媒体: ○○）

- 幹: 「……」
- 対象 / 持ち帰り: ……（避けたいこと: ……）
- 型: ……
- 流れ: 1.…… → 2.…… → 3.……（ハイライト: N）
- 削った: ……（退避先: ……）
```

## 振る舞いのコツ

### やってはいけないこと

- **文章レベルの推敲に踏み込まない** — 語彙・リズムの直しは別タスク（小説なら [novel-revision](../novel-revision/SKILL.md)）
- **各媒体の記法ルールを変えない** — 小説記法は [35-novels.md](../../rules/35-novels.md)、Slidev は [10-tech-stack.md](../../rules/10-tech-stack.md) が正
- **観点を一気に全部出さない** — 2-3 個ずつ
- **物語論用語を転用例なしで使わない**

### やったほうがいいこと

- **観点を名指しする** — 「これは焦点化（観点 5）の揺れです」と用語で共有
- **論点にトレードオフを 1 行添える** — 「a は方向性の判断、b は機械的に直せる」
- **媒体アダプタの実例を引く** — スライドなら [_slide_structure.md](../../../src/content/slides/2026-05-15-cloudnative-kaigi/_slide_structure.md)、小説なら [_plot.md](../../../src/content/novels/_shared/_templates/_plot.md)
- **ユーザーが手で決めた構成を尊重する** — 「ここはこの順でいく」と決めた箇所を勝手に組み替えない
